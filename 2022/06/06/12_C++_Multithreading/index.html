#12 C++多线程

2022-06-06
  • C++

概述

C++11之前，C++的多线程编程均需依赖系统或第三方接口实现，一定程度上影响了代码的移植性。

C++11中引入了boost库中的多线程部分内容，形成C++标准。

成为C++标准后的boost多线程编程部分接口基本保持不变，方便以前使用boost接口开发的使用者切换升级。

头文件

C++11新标准中引入了五个头文件来支持多线程编程：

<atomic>：主要声明了两个类atomic和atomic_flag，还声明了一套 C 风格的原子类型和与 C 兼容的原子操作的函数。

<thread>：主要声明了thread类，this_thread命名空间也在该头文件中。

<mutex>：主要声明了与mutex相关的类，包括mutex系列类，lock_guard，unique_lock以及其他的类型和函数。

<condition_variable>：主要声明了与条件变量相关的类，包括condition_variable和condition_variable_any。

<future>：主要声明了promise，package_task两个Provider类，以及future和shared_future两个Future类，另外还有一些与之相关的类型和函数，如async()函数。condition_variable和future在vc/clr下不支持。

Thread类

Main函数创建主进程，同时具备一个主线程。任何线程都可以通过Thread类创建子线程。

Thread th; 默认构造函数，不与任何执行线程关联。

通过给构造函数指定一个函数，线程对象关联一个执行线程，该线程执行该函数。

#include <iostream>
#include <thread>
using namespace std;

int f1() {
    cout << "in sub thread" << endl;
    return 1;
}

int main() {
    cout << "main thread " << endl;
    thread th1(f1);
    th1.join();  // 消除异常
    //若无该语句则输出
    //main thread
    //tierminate called without an active exception
    //n sub thread
}

线程id

每个线程都具有唯一的thread::id标识符。

线程自身可通过this_thread::get_id返回其ID。

调用该子线程的线程（简称调用线程）可通过thread::get_id返回子线程的ID。

native_handle;是由系统分配的ID，二者不同。

#include <iostream>     // std::cout
#include <thread>       // std::thread
using namespace std;

int f1() {
    cout << "in sub thread : " << this_thread::get_id() << endl;
    return 1;
}

int main() {
    cout << "main thread: " << this_thread::get_id() << endl;
    thread th1(f1);
    cout << "sub thread id:" << th1.get_id() << endl;
    cout << "sub thread native handle" << th1.native_handle() << endl;
    th1.join();
}
//可能输出
//main thread: 50604
//sub thread id:54168
//sub thread native handle00000000000000B4
//in sub thread : 54168

Join与Detach

子线程一旦创建，即刻分道扬镳，主线程与子线程各自独立运行。

哪个线程执行的时间长，未知。但主线程终止后，进程会终止，从而强制终止所有子线程，但子线程分配的资源可能不能释放。

在主线程与子线程之间定义两种关系：

  • 可会合的（joinable）

主线需要显式等待子线程结束，即使子线程本来就比主线程更快结束。否则，子线程的资源不能被回收。主线程调用子线程.join()来等待其结束。

  • 分离的（detached）

子线程与主线程彻底分家，一旦子线程结束，系统会自动释放其资源。主线程调用子线程.detach()来使之分离。

默认的子线程都是可会和的,调用detach()使之分离，但反向操作不行，不能由detached变为joinable。

所有的子线程，在主线程中，要么join之，要么detach之。如前例中th1.detach();

this_thread名称空间

std名称空间下的一个子空间：

void f(int i) {
    cout << i << endl;
    cout << this_thread::get_id() << endl;
    this_thread::sleep_for()
    this_thread::sleep_until()
    this_thread::yield();
}

this_thread::yield()：避免一个线程频繁与其他线程争抢CPU时间片，从而导致多线程处理性能下降。

比如需要等待某个操作完成，如果直接用一个循环不断判断这个操作是否完成，会使得这个线程占满CPU时间，造成资源浪费。

可以判断一次操作是否完成，如果没有完成就调用yield交出时间片，过一会儿再来判断是否完成，线程占用CPU时间会大大减少。

举例：

while(!isDone());  //不好
while(!isDone()) yield();   // 稍好

Mutex头文件

mutex：最基本也是最常用的互斥类。

recursive_mutex：同一线程内可递归（重入）的互斥类。

timed_mutex ：除具备mutex功能外，还提供了带时限请求锁定的能力。

recursive_timed_mutex：同一线程内可递归（重入）的timed_mutex。

Mutex标准操作

lock

lock：锁住互斥量。调用lock时有三种情况:

如果互斥量没有被锁住，则调用线程将该mutex锁住，直到调用线程调用unlock释放。

如果mutex已被其它线程lock，则调用线程将被阻塞，直到其它线程unlock该mutex。

如果当前mutex已经被调用者线程锁住，则std::mutex死锁，而recursive系列则成功返回。

try_lock

try_lock尝试锁住mutex，调用该函数同样也有三种情况：

如果互斥量没有被锁住，则调用线程将该mutex锁住（返回true），直到调用线程调用unlock释放。

如果mutex已被其它线程lock，则调用线程将失败，并返回false。

如果当前mutex已经被调用者线程锁住，则std::mutex死锁，而recursive系列则成功返回true。

unlock

unlock负责解锁mutex，释放对mutex的所有权。

对于recursive系列mutex，unlock次数需要与lock次数相同才可以完全解锁。

#include <iostream>
#include <thread>
using namespace std;

void badinc(int loop, int *counter) {
    for (int i = 0; i < loop; i++)
        ++*counter;
}

int main() {
    thread threads[5];
    int counter = 0;
    for (thread &thr : threads)
        thr = thread(badinc, 10000, &counter);
    for (thread &thr : threads)
        thr.join();
    cout << counter << endl;
    return 0;
}
//每次输出各不相同 均小于5w 因线程未同步
//出人意料的是gcc居然能直接输出5w amazing

#include <iostream>
#include <thread>
#include <mutex>
using namespace std;

void goodinc(mutex *pmutex, int loop, int *counter) {
    for (int i = 0; i < loop; i++) {
        pmutex->lock();
        ++*counter;
        pmutex->unlock();
    }
}

int main() {
    thread threads[5];
    mutex mutex;
    int counter = 0;
    for (thread &thr : threads)
        thr = thread(goodinc, &mutex, 10000, &counter);
    for (thread &thr : threads)
        thr.join();
    cout << counter << endl;
    return 0;
}
//同步处理后结果为5w

RAII

RAII，Resource Acquisition is Initialization，即资源获取即初始化，由Bjarne Stroustrup提出。

std::mutex mutex_;

void function() {
    mutex_.lock();
    ......
    ......
    mutex_.unlock();
}

在互斥量lock和unlock之间的代码很可能会出现异常，或者有return语句，互斥量就不会正确的unlock，会导致线程死锁。

更好的方式是使用std::unique_lock或者std::lock_guard对互斥量进行状态管理：

mutex mutex_;

void function() {
    lock_guard<mutex> lock(mutex_);
    ......
    ......
}

在创建lock_guard对象的时候，会对mutex对象进行lock，当std::lock_guard对象在超出作用域时，会自动std::mutex对象进行解锁，不用担心代码异常造成线程死锁。

lock_guard类

lock_gurad是C++11中定义的模板类。

template <class Mutex> class lock_guard;

类lock_guard是互斥封装器，为在作用域块期间占有互斥提供了一种方便的RAII风格机制。

创建lock_guard对象时，它试图接收给定互斥的所有权。控制离开创建lock_guard对象的作用域时，销毁lock_guard并释放互斥。

lock_guard类不可复制。

lock_guard对象并不负责管理Mutex对象的生命周期，lock_guard对象只是简化了Mutex对象的上锁和解锁操作，方便线程对互斥量上锁，即在某个lock_guard对象的声明周期内，它所管理的锁对象会一直保持上锁状态；而当lock_guard的生命周期结束之后，它所管理的锁对象会被解锁。

互斥锁保证了线程间的同步，但是却将并行操作变成了串行操作，这对性能有很大的影响，所以我们要尽可能的减小锁定的区域，也就是使用细粒度锁。

lock_guard不够灵活，它只保证在析构的时候执行解锁操作，lock_guard本身并没有提供加锁和解锁的接口，但是有些时候会有这种需求。

class LogFile {
        mutex _mu;
        ofstream f;
    public:
        LogFile() {
            f.open("log.txt");
        }
        ~LogFile() {
            f.close();
        }
        void shared_print(string msg, int id) {
            {
                std::lock_guard<std::mutex> guard(_mu);
                //do something 1
            }
            //do something 2
            {
                std::lock_guard<std::mutex> guard(_mu);
                // do something 3
                f << msg << id << endl;
                cout << msg << id << endl;
            }
        }
};

Shared_print函数内部有两段代码需要进行保护，这个时候使用lock_guard就需要创建两个局部对象来管理同一个互斥锁。当然也可以只创建一个，但是锁的力度太大，效率不高。

可以代替使用unique_lock。它提供了lock()和unlock()函数，能记录现在处于上锁还是没上锁状态，在析构的时候，会根据当前状态来决定是否要进行解锁（lock_guard就一定会解锁）。

class LogFile {
        std::mutex _mu;
        ofstream f;
    public:
        LogFile() {
            f.open("log.txt");
        }
        ~LogFile() {
            f.close();
        }
        void shared_print(string msg, int id) {

            std::unique_lock<std::mutex> guard(_mu);
            //do something 1
            guard.unlock(); //临时解锁

            //do something 2
            guard.lock(); //继续上锁
            // do something 3
            f << msg << id << endl;
            cout << msg << id << endl;
            // 结束时析构guard会临时解锁
            // 这句话可要可不要，不写，析构的时候也会自动执行
            // guard.ulock();
        }

};
//在无需加锁的操作时，可以先临时释放锁，然后需要继续保护的时候，可以继续上锁，这样就无需重复地实例化lock_guard对象，还能减少锁的区域。

可以使用defer_lock设置初始化unique_lock的时候不进行默认的上锁操作：

void shared_print(string msg, int id) {
    std::unique_lock<std::mutex> guard(_mu, std::defer_lock);
    //do something 1

    guard.lock();
    // do something protected
    guard.unlock(); //临时解锁

    //do something 2

    guard.lock(); //继续上锁
    // do something 3
    f << msg << id << endl;
    cout << msg << id << endl;
    // 结束时析构guard会临时解锁
}

unique_lock

unique_lock对象以独占所有权的方式（unique owership）管理mutex对象的上锁和解锁操作。

unique_lock对象也不负责管理Mutex对象的生命周期，只是简化了Mutex对象的上锁和解锁操作，方便线程对互斥量上锁，即在某个unique_lock对象的生命周期内，它所管理的锁对象会一直保持上锁状态；而unique_lock的生命周期结束之后，它所管理的锁对象会被解锁，这一点和lock_guard类似，但unique_lock提供了更多的自由。

unique_lock比lock_guard灵活。但因为它内部需要维护锁的状态，所以效率比lock_guard低。在不能用lock_guard解决问题的时候，才使用unique_lock。

条件变量

单独使用互斥有时并不能解决全部问题。

生产者消费者模型

假设想实现一个简单的消费者生产者模型：一个线程往队列中放入数据（假定生产过程比较缓慢）；一个线程往队列中取数据，须确保队列中有数据；这个队列由线程间共享；需要使用互斥锁对队列进行保护，一个线程在往队列添加数据的时候，另一个线程不能取，反之亦然。

用互斥锁实现如下：

#include <iostream>
#include <thread>
#include <mutex>
#include <deque>
using namespace std;
deque<int> q;
mutex mu;

void produce() {
    int count = 10;  // 一共产生10个数字。
    while (count > 0) {
        unique_lock<mutex> locker(mu);
        q.push_front(count);  // 从前面压入
        locker.unlock();
        this_thread::sleep_for(chrono::milliseconds(500));
        // 用以模拟耗时的任务。
        count--;
    }
}

void consume() {
    int data = 0;
    while (data != 1) { // 循环读到最后一个数字
        unique_lock<mutex> locker(mu);
        if (!q.empty()) {
            data = q.back();
            q.pop_back(); // 从后面弹出
            locker.unlock();
            cout << "comsumer got a value from producer: " << data << std::endl;
        } else {
            locker.unlock();
            //this_thread::sleep_for(chrono::milliseconds(500)); 添加延时
        }
    }
}

int main() {
    thread t1(produce);
    thread t2(consume);
    t1.join();
    t2.join();
    return 0;
}
//依次蹦出comsumer got a value from producer:10~1

在生产过程中，this_thread::sleep_for(chrono::milliseconds(500));表示延时0.5s，以模拟缓慢的生产过程。

消费函数中while循环，直至最后一个数字才会停止，每次循环内部，先加锁，判断队列情况：如不空，就取出一个数；如空，直接解锁。在1s的生产时间内，消费者不断地while循环试图消费，做了很多无用功。

CPU占用率高。在Surface book 2 上测试CPU占据约10%（我也不知道老师这怎么测的，我是测不出hh）

解决办法之一是给消费者也加一个小延时，如果一次判断后，发现队列是空的，就延时500ms，这样可以减小CPU的占用率。

然而困难之处在于，生产者的延时有时未知，甚至是随机的。消费者固定延时500ms，有可能无谓的等待，（如果生产者延时短），有可能还是不必要的占用了CPU时间（即while循环），如果生产者延时很长长长。

条件变量（Condition Variable）

C++11头文件<condition_variable>，提供了condition_variable类。

condition_variable可与mutex结合一起使用，其中有两个重要的函数，

wait()：可以让线程陷入休眠状态。在消费者生产者模型中，如果消费者发现队列中没有东西，就可以让自己休眠，

notify_one()：唤醒处于wait中的一个条件变量（可能当时有很多条件变量都处于wait状态）。 当生产者往队列中放入数据之后，使用notify_one()的可以叫醒等待中的线程。

#include <iostream>
#include <thread>
#include <mutex>
#include <deque>
#include <condition_variable>
using namespace std;
deque<int> q;
mutex mu;
condition_variable cond;

void produce() {
    int count = 10;
    while (count > 0) {
        unique_lock<mutex> locker(mu);
        q.push_front(count);
        locker.unlock();      //a释放锁
        cond.notify_one();  //b通过条件变量唤醒其他等待线程
        this_thread::sleep_for(chrono::seconds(1));
        count--;
    }
}

void consume() {
    int data = 0;
    while (data != 1) { //循环读到最后一个数字
        unique_lock<mutex> locker(mu);
        while (q.empty())
            cond.wait(locker); //将mutex解锁，并进入休眠，等待唤醒
        data = q.back();  //唤醒后，立刻拥有锁
        q.pop_back();     //读取数据后，
        locker.unlock();   //释放锁。
        cout << "comsumer got a value from producer: " << data << std::endl;
    }
}

int main() {
    thread t1(produce);
    thread t2(consume);
    t1.join();
    t2.join();
    return 0;
}

//在notify_one()的时候，不需要处于互斥锁的保护范围内，所以在唤醒条件变量之前可以将锁unlock()
//此例中a,b行可以互换

在管理互斥锁的时候，必须使用unique_lock而不是lock_guard

wait()函数所做的事情：在wait()函数之前，使用互斥锁加锁了。wait()函数会先调用互斥锁的unlock()函数。（自己马上要睡眠，交出互斥锁，让其他线程可以工作）然后再将自己睡眠，在被唤醒后，又会继续持有锁，保护后面的队列操作。而lock_guard没有lock和unlock函数，所以不能使用。

while (q.empty())
    cond.wait(locker); // Unlock mu and wait to be notified

在consumer中，在判断队列是否为空的时候，使用的是while(q.empty())，而不是if(q.empty())，这是因为wait()从阻塞到返回，不一定就是由于notify_one()函数造成的，还有可能由于系统的不确定原因唤醒（可能和条件变量的实现机制有关），这个的时机和频率都是不确定的，被称作伪唤醒，如果在错误的时候被唤醒了，执行后面的语句就会错误，所以需要再次判断队列是否为空，如果还是为空，就继续wait()阻塞。

上一段代码还可以换一种写法：

wait()的第二个参数可以传入一个函数表示检查条件，这里使用lambda表达式最为简单，如果这个函数返回的是true，wait()函数不会阻塞会直接返回，如果这个函数返回的是false，wait()函数就会阻塞着等待唤醒，如果被伪唤醒，会继续判断函数返回值。

cond.wait(locker, []() {return !q.empty(); });

除了notify_one()函数，C++还提供了notify_all()函数，可以同时唤醒所有处于wait状态的条件变量。

void consume() {
    int data = 0;
    while ( data != 1) {
        std::unique_lock<std::mutex> locker(mu);
        cond.wait(locker, []() {
            return !q.empty();
        } );  // Unlock mu and wait to be notified
        data = q.back();
        q.pop_back();
        locker.unlock();
        std::cout << "t2 got a value from t1: " << data << std::endl;

单生产者单消费者模型

#include <cstdlib>
#include <condition_variable>
#include <iostream>
#include <mutex>
#include <thread>
using namespace std;

static const int kItemRepositorySize = 10; // 库房容量
static const int kItemsToProduce = 1000;   // 生产1000个产品.
struct ItemRepository {
    int item_buffer[kItemRepositorySize]; // 产品缓冲区, 配合 read_position 和 write_position 模型环形队列.这里一个产品就是一个整数
    size_t read_position; // 消费者读取产品位置.
    size_t write_position; // 生产者写入产品位置.
    mutex mtx; // 互斥量,保护产品缓冲区
    condition_variable repo_not_full; // 条件变量, 指示产品缓冲区不为满.
    condition_variable repo_not_empty; // 条件变量, 指示产品缓冲区不为空.
} gItemRepository; // 产品库全局变量, 生产者和消费者操作该变量.

void ProduceItem(ItemRepository *ir, int item) {
    unique_lock<mutex> lock(ir->mtx);
    while (((ir->write_position + 1) % kItemRepositorySize)
            == ir->read_position) { //读位置追上了写位置，库房满。.
        cout << "生产者等待...\n";
        (ir->repo_not_full).wait(lock); // 生产者等待"产品库缓冲区不为满"这一条件发生.
    }
    (ir->item_buffer)[ir->write_position] = item; // 写入产品.
    (ir->write_position)++; // 写入位置后移.
    if (ir->write_position == kItemRepositorySize) // 写入位置若是在队列最后则重新设置为初始位置.
        ir->write_position = 0;
    (ir->repo_not_empty).notify_all(); //  A1 通知消费者产品库不为空.
    lock.unlock(); // B1 解锁.
    // A1,B1 两句位置可交换
}

int ConsumeItem(ItemRepository *ir) {
    int data;
    unique_lock<mutex> lock(ir->mtx);
    while (ir->write_position == ir->read_position) {// 库房为空
        cout << "消费者等待...\n";
        (ir->repo_not_empty).wait(lock); // 消费者等待"产品库缓冲区不为空"这一条件发生.
    }
    data = (ir->item_buffer)[ir->read_position]; // 读取某一产品
    (ir->read_position)++; // 读取位置后移
    if (ir->read_position >= kItemRepositorySize) // 读取位置若移到最后，则重新置位.
        ir->read_position = 0;
    (ir->repo_not_full).notify_all(); //A2 通知生产者产品库不为满.
    lock.unlock(); //B2  解锁.
    // A2,B2 两句位置可交换
    return data; // 返回产品.
}

void ProducerTask() { // 生产者任务
    for (int i = 1; i <= kItemsToProduce; ++i) {
        cout << "生产第 " << i << "件产品..." << endl;
        ProduceItem(&gItemRepository, i); // 循环生产 kItemsToProduce 个产品.
    }
}

void ConsumerTask() { // 消费者任务
    static int cnt = 0;
    while (1) {
        int item = ConsumeItem(&gItemRepository); // 消费一个产品.
        cout << "消费第 " << item << "件产品" << endl;
        if (++cnt == kItemsToProduce)
            break; // 如果产品消费个数为 kItemsToProduce, 则退出.
    }
}

void InitItemRepository(ItemRepository *ir) {
    ir->write_position = 0; // 初始化产品写入位置.
    ir->read_position = 0; // 初始化产品读取位置.
}

int main() {
    InitItemRepository(&gItemRepository);
    thread producer(ProducerTask); // 创建生产者线程.
    thread consumer(ConsumerTask); // 创建消费之线程.
    producer.join();
    consumer.join();
}

多生产者多消费者模型

#include <cstdlib>
#include <condition_variable>
#include <iostream>
#include <mutex>
#include <thread>
using namespace std;
static const int kItemRepositorySize = 10; // 库房容量.
static const int kItemsToProduce = 1000;   // 产品数量.

struct ItemRepository {
    int item_buffer[kItemRepositorySize]; // 环形缓冲区库房
    size_t read_position;
    size_t write_position;
    size_t produced_item_counter;
    size_t consumed_item_counter;
    mutex mtx;
    mutex produced_item_counter_mtx;  // 生产计数器互斥
    mutex consumed_item_counter_mtx;  // 消费计数器互斥
    condition_variable repo_not_full;
    condition_variable repo_not_empty;
} gItemRepository;

void ProduceItem(ItemRepository *ir, int item) {
    unique_lock<mutex> lock(ir->mtx);
    while (((ir->write_position + 1) % kItemRepositorySize)
            == ir->read_position) { // 库房已满.
        cout << "生产者等待...\n";
        (ir->repo_not_full).wait(lock);
    }
    (ir->item_buffer)[ir->write_position] = item;
    (ir->write_position)++;
    if (ir->write_position == kItemRepositorySize)
        ir->write_position = 0;
    (ir->repo_not_empty).notify_all();// A1. 唤醒所有消费者等待线程
    lock.unlock();// B1. 解锁
    // A1,B1 可以交换位置。
}

int ConsumeItem(ItemRepository *ir) {
    int data;
    unique_lock<mutex> lock(ir->mtx);
    // 库房为空
    while (ir->write_position == ir->read_position) {
        cout << "消费者等待...\n";
        (ir->repo_not_empty).wait(lock);
    }
    data = (ir->item_buffer)[ir->read_position];
    (ir->read_position)++;
    if (ir->read_position >= kItemRepositorySize)
        ir->read_position = 0;
    (ir->repo_not_full).notify_all();// A2. 唤醒所有生产者等待线程
    lock.unlock();  // B2. 解锁
    // A2,B2 可以交换位置。
    return data;
}

void ProducerTask() {
    bool ready_to_exit = false;
    while (1) {
        unique_lock<mutex> lock(gItemRepository.produced_item_counter_mtx);
        // 生产数量计数器
        if (gItemRepository.produced_item_counter < kItemsToProduce) {
            ++(gItemRepository.produced_item_counter);
            ProduceItem(&gItemRepository, gItemRepository.produced_item_counter);
            cout << "生产者线程 " << this_thread::get_id()
                 << " 正在生产第 " << gItemRepository.produced_item_counter
                 << "件产品" << endl;
        } else
            ready_to_exit = true;
        lock.unlock();
        if (ready_to_exit == true)
            break;
    }
    cout << "生产者线程 " << this_thread::get_id()
         << " 正退出..." << endl;
}

void ConsumerTask() {
    bool ready_to_exit = false;
    while (1) {
        unique_lock<mutex> lock(gItemRepository.consumed_item_counter_mtx);
        // 消费数量计数器
        if (gItemRepository.consumed_item_counter < kItemsToProduce) {
            int item = ConsumeItem(&gItemRepository);
            ++(gItemRepository.consumed_item_counter);
            cout << "消费者线程 " << this_thread::get_id()
                 << " 正在消费第 " << item << "件产品" << endl;
        } else
            ready_to_exit = true;
        lock.unlock();
        if (ready_to_exit == true)
            break;
    }
    cout << "消费者线程 " << this_thread::get_id()
         << " 正退出..." << endl;
}


void InitItemRepository(ItemRepository *ir) {
    ir->write_position = 0;
    ir->read_position = 0;
    ir->produced_item_counter = 0;
    ir->consumed_item_counter = 0;
}

int main() {
    InitItemRepository(&gItemRepository);
    thread producer1(ProducerTask);
    thread producer2(ProducerTask);
    thread producer3(ProducerTask);
    thread producer4(ProducerTask);
    thread consumer1(ConsumerTask);
    thread consumer2(ConsumerTask);
    thread consumer3(ConsumerTask);
    thread consumer4(ConsumerTask);
    producer1.join();
    producer2.join();
    producer3.join();
    producer4.join();
    consumer1.join();
    consumer2.join();
    consumer3.join();
    consumer4.join();
}
